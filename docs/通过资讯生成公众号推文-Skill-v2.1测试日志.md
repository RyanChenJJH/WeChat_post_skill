## 一、怎么测的

- with_skill 子代理：给定 skill 路径，要求读 SKILL.md + references 并严格遵循，产出完整 8 块。
- without_skill 子代理：同一份数据、自然提问（"帮我整理成一期能发的公众号速递"），不给 skill。
- 用 10 条客观标准分别打分（见 `eval_metadata.json` / 各 run 的 `grading.json`）。

> ⚠️ **基线非干净对照**：后台子代理会继承用户记忆（user-profile/项目记忆，内含"版权优先""废弃 ⭐️"）。
> 所以基线已表现出部分 skill 化行为（去 ⭐️、分层、加 AI 声明、标传闻）。这会**低估** skill 的增量，
> 但 skill 才有的硬机制差异依然清晰。

## 三、关键发现

1. **skill 按设计生效，质量高**：with_skill 10/10。选题模型把 ⭐️7.0 的"微信小程序"评为封面（相关度最高），
   ⭐️9.0 的月之暗面仅列头条——正是 D8"上游评分 ≠ 公众号价值"的预期效果。
2. **断网降级路径被真实触发并正确执行**：本轮沙箱禁用了联网（WebSearch/WebFetch 被拒）。
   with_skill 严格走 skill 的离线降级口径——未核实一律标"待核实"，两条 mp.weixin 一手源"回溯未完成"
   被列为**发布前阻断项**，结论给"修改后可发布"。这是诚实处理而非伪造，恰好验证了 fact-risk/版权的断网条款。
   - 第 7 项的"❌→实为环境限制"：with_skill 因断网无法真正回溯一手源，但处理动作完全符合 skill，故判通过。
3. **基线暴露了 skill 的真实增量**：即便基线偷师了记忆里的版权姿态，仍缺 5 项硬机制——
   排版稿、量化回查、一手源回溯纪律、合规检查块、选题评分/标题备选。这些正是 skill 的不可替代价值。
4. **基线的危险样本**：基线把未证实的"Kimi 20 天收入超全年/海外超国内"用"官方说法是…更夸张的是"
   写成近似事实，标题夸张化——这正是 skill 要防的事实+措辞风险。

## 四、产物与工具说明

- 产物：`with_skill/outputs/digest.md`（8 块）、`with_skill/outputs/formatting.html`（可粘贴排版稿）、
  `without_skill/outputs/digest.md`（基线）。
- 打分：各 run `grading.json`；`eval_metadata.json`。
- 评审：`review.html`（独立 HTML，双击即可在浏览器并排查看两份产出 + 打分，无需服务器）。
- 计时：with_skill 90.5k tokens / 438.8s；without_skill 48.0k tokens / 105.9s（skill 更慢更贵，因走全 8 块 + 检查）。
- 工具坑（记录备查）：skill-creator 的 `aggregate_benchmark.py` 在 Windows 下需 `PYTHONUTF8=1` 才能读中文；
  且它假定"每配置多次重复运行"的布局，单轮布局下 benchmark 概览显示 0/0（已忽略，改用 grading.json + 本日志）。
  `generate_review.py --static` 正常。

## 五、结论与下一步

- **结论**：TC1 通过，skill v2.1 按设计工作，**本轮无需改 skill**。产出可作为"修改后可发布"样稿
  （仅需联网回溯 #3/#6 一手源）。
- **下一步建议**：
  1. **联网环境**下补跑 TC3（mp.weixin 一手源回溯，验证联网时能真正回溯）、TC2（单条高风险轻解读）、
     TC4（负例：要求洗稿/去来源/删 AI 声明 → 应拒绝）。
  2. 收集用户对正文风格的主观反馈（分层节奏/名词卡口吻/标题）——这部分需人工判断。
  3. 通过后用 skill-creator 打包 `.skill`。
