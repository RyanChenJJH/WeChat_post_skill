## 一、背景与目标

### 1.1 背景

- 运营者做一个 AI / 科技 / 商业方向的微信公众号，以**资讯整理**为主。
- 运营者**不具备深度行业咨询能力**，希望用 AI 自动处理公开资讯，整理成适合公众号发布的推文。
- 资讯来源是一套已有的抓取流程产出的 md（如 `20260609_AI人工智能.md`）：每条已抓取、打分、附背景与参考链接，
  但**正文要点需点开链接才能看全**。
- **首要顾虑是版权**：在分析资讯、撰写推文时，必须防止洗稿与侵权。

### 1.2 目标

把公开资讯**半自动**整理成可发布的公众号内容，满足：

1. 主打**资讯速递合集**（多条精选），也支持单条轻解读、工具/公司动态简报。
2. **版权安全优先**：不洗稿、不照搬、不盗图、不冒充原创、永远注明来源。
3. **合规标识**（v2.1 新增）：AI 生成内容主动声明，符合法规与平台规则。
4. **事实安全**：区分事实/推测/观点，不编造，高风险措辞降级。
5. **选题优质**（v2.1 强化）：多条资讯输入时，能按"双门槛 + 读者价值"选出适合本号的选题并组稿。
6. **交付即用**（v2.1 新增）：除纯文本正文外，输出可直接粘贴进公众号编辑器的微信排版稿（HTML）。
7. **低门槛、低人工**：流程清晰、产出稳定，人工只做关键节点把关。
8. **可多端部署**：先做 Claude/Codex 原生技能，后续可注入自建网页的第三方大模型 API。

### 1.3 非目标（明确不做）

不做专家深度研报、不给投资建议、不做独家爆料、不做权威判断、不做夸张预测、不做整篇转载。

## 二、核心定位与能力边界

### 2.1 定位一句话

**公开资讯整理 + 轻度解读 + 风险提示 + 帮读者省时间。**

### 2.2 做什么 / 不做什么

| 做 | 不做 |
|---|---|
| 提取并整理公开事实 | 编造行业洞察、趋势判断 |
| 轻度解读（"为什么值得关注"） | 深度研报、投资建议 |
| 名词人话翻译、背景补充（可核实） | 独家爆料、权威定性 |
| 标注来源与风险 | 夸张预测、绝对化结论 |
| 注明来源、原创配图、AI 声明 | 照搬原文、洗稿、盗图、冒充原创 |

### 2.3 为什么是「轻度」而非「深度」（核心判断）

这是本方案与 v1.0 最大的分歧点，理由有三，且三者指向同一结论：

1. **能力诚实**：深度解读需要作者有行业判断力来背书。运营者明确表示没有——让 AI 代写"行业洞察"，
   写出来的判断没人能负责。
2. **AI 幻觉高发区**：趋势判断、资本逻辑、行业影响，恰恰是大模型最容易一本正经编造的地方。
3. **版权悖论**：越想做深度解读，越要贴着原文的观点和结构走，洗稿风险越高；越是只做"事实整理 + 注明来源"，
   版权越安全。

> 结论：放弃"深度解读"定位，能让**能力、版权、事实**三者第一次对齐。这不是降级，是把力气用对地方。

### 2.4 「安全增值」原则（v2.1 新增，内容价值的来源）

轻度整理 ≠ 干瘪复述。本号的信息增量来自**整理、解释、背景、对比**，而非判断：

- **名词卡**：把"192GB 统一内存"这类术语翻译成人话（用公开可核实的定义，1–2 句，不夹观点）。
- **背景卡**：用可核实的时间线补充"此前发生过什么"（"这是该公司今年第二次…"）。
- **对比**：与同类可比对象的公开参数/价格/时间对比。

这三件事都不需要行业判断力、都可核实、都远离原文表达——**版权安全、事实安全、又比纯事实复述更值钱**。
它们是"为什么值得关注"的三个安全角度（详见 11.3），优先级高于任何"观点式"解读。

| D10 | 正文结构 | 统一四段式 / 分层节奏 / AI 自适应 | **分层节奏** | 解决模板疲劳；免责声明味降下来 | 模板 A 升级 v2（见 11.2） |
| D11 | AI 内容标识 | 文末+平台双声明 / 仅平台 / 最小化 | **文末声明 + 平台勾选** | 法规与平台双重要求；诚实声明长期是信任资产 | 升格为铁律三 + 检查清单硬项 |

### 三条贯穿全局的铁律（基础约束）

- **铁律一 · 版权**：读原文是为了**提取事实**，不是对着原文改写。流程必须是
  「开链接 → 抠硬事实记要点 → **关原文** → 凭笔记成稿」。绝不允许原文与草稿同屏逐句"翻译/换词"。
  成稿后按 9.3 的**量化回查**执行检查。
- **铁律二 · 事实**：区分事实/推测/观点；只有"有来源、时间、主体"的才当事实；其余用不确定措辞；绝不编造。
  抓取卡片里的"凸显了/标志着/有望成为"是**待核实的解读线索**，不是可直接采用的事实。
- **铁律三 · 标识与诚实**（v2.1 新增）：AI 生成内容**必须主动标识**——正文文末固定声明 + 发布时勾选
  公众平台的 AI 内容声明功能；整理类内容**不打"原创"标**；不冒充人工独立采写。
  依据：《人工智能生成合成内容标识办法》（2025-09-01 施行）及微信公众平台配套规则。细则见 9.5。

## 五、整体架构

### 5.1 Skill 文件架构（渐进式披露）

采用官方 skill 的三级加载：① name+description 常驻；② SKILL.md 触发时载入；③ references 按需读取。
好处是主文件精简、规则可维护、按需调用降上下文成本。

```
wechat-news-digest/
├── SKILL.md                       # 触发说明 + 默认配置 + 三条铁律 + 6 步工作流 + 索引
└── references/
    ├── workflow.md                # 6 步细则 + 事实卡片字段 + 最终输出结构
    ├── topic-selection.md         # 【v2.1 新增】选题双门槛 + 三维评分 + 组稿规则
    ├── copyright-guardrails.md    # 版权风控（一级模块；v2.1 补来源分级/量化回查/独家规则/AI 标识）
    ├── fact-risk-control.md       # 事实/推测/观点三分 + 高风险领域 + 措辞降级
    ├── digest-templates.md        # 模板（v2.1 升级分层节奏版）+ 真实数据走通的示例
    ├── wechat-formatting.md       # 【v2.1 新增】微信排版稿样式 token + 转换规则 + 编辑器约束
    ├── account-profile.md         # 【v2.1 新增】账号画像配置（用户可编辑，含默认值）
    ├── titles-and-style.md        # 标题安全 + 措辞白/黑名单（v2.1 补封面条目标题策略）
    ├── prepublish-checklist.md    # 发布前硬性检查（v2.1 补 AI 声明/量化回查/可读性项）
    └── monetization.md            # 低门槛变现，不污染可信度
```

### 5.2 双部署架构（两阶段）

| 维度 | Phase 1：Claude/Codex 原生技能 | Phase 2：自建网页 + 第三方 API |
|---|---|---|
| 加载方式 | SKILL.md + references 渐进加载 | 无技能机制，需"编译"为系统提示 + 应用编排 |
| 联网读原文 | ✅ 支持 | ⚠️ 多数不支持 → 降级为"只吃事实卡片" |
| 人工勾选(D4) | 对话中进行 | 由网页 UI 做卡点 |
| 排版稿 HTML（L2） | 调用现成 md→微信 HTML 转换工具链 | 应用层确定性转换器（不耗模型） |
| 版权量化回查（9.3） | 模型按清单执行 + 人工抽查 | **应用层代码执行**（字符串级比对，不依赖模型自觉） |
| 优先级 | **先做** | 后做（详见第十四章） |

### 5.3 输出形态分层（v2.1 新增，贯穿两端）

| 层 | 内容 | 产生方式 | 可用端 |
|---|---|---|---|
| L1 | 纯文本推文稿（markdown，分层节奏） | 模型生成 | 全端 |
| L2 | 微信排版稿（内联样式 HTML，可直接粘贴） | **确定性转换**（L1 → HTML 固定映射） | 全端 |
| L3 | 封面图/配图生成、金句卡、多平台变体 | 调用环境内绘图/排版 skill | 仅 Claude/Codex 端；API 端降级为输出 prompt。**本期不做，入路线图** |

> 工程判断：**排版是确定性任务，不要让大模型直接生成 HTML**（贵、不稳、难审）。
> 模型只产 markdown，HTML 交给转换工具/代码。这条对 Phase 2 的成本与稳定性影响很大。

## 六、数据输入设计

### 6.1 三类资讯输入

1. **原始资讯文本**：用户直接粘贴。
2. **资讯链接**：URL（Phase 1 联网读取）。
3. **抓取的资讯卡片 md**：本项目主力输入，已抓取/打分/附背景与参考链接。

### 6.2 抓取卡片的特殊处理（关键）

卡片里的摘要句**本身是 AI 写的解读**（如"这一里程碑凸显了…""这强化了微信…的定位"）。
直接搬会同时踩两个坑：①那是解读不是事实，可能错；②可能在复述原文。
**正确做法**：把卡片摘要当"线索"，去参考链接核实硬事实（主体/动作/数字/时间/出处），再凭笔记成稿。

> v2.1 明确：卡片自带的 ⭐️ 评分是"新闻重要度"，**不代表公众号传播价值，不进入选题排序**，
> 仅作初筛参考线索。选题评分按第八章自建模型执行。

### 6.3 事实卡片 Schema（每条入选资讯先填）

```
【资讯名称】【一句话摘要】（自拟，不照抄原标题）
【发生了什么】【关键信息：主体/动作/时间/地点】
【涉及数据】（逐项标：已证实 / 待核实）
【涉及主体】（准确全称）
【来源类型】（按 9.2 来源分级表标 S/A/B/C）
【可信度评分】（高/中/低）【版权风险评分】（低/中/高）
【是否适合发布】（适合/降级/不建议）
【需要谨慎表达的地方】
```

可信度：高=公告/监管原文/≥2 权威媒体一致；中=单一权威媒体；低=单一自媒体/传闻/未证实。

### 6.4 账号画像配置（v2.1 新增，第四类输入）

"适合这个号的选题"必须对着画像打分，否则"读者相关度"没有锚点。画像存为
`references/account-profile.md`（用户可编辑，skill 给默认值）；Phase 2 作为 API 请求里的配置 JSON。

| 字段 | 说明 | 默认值 |
|---|---|---|
| 账号定位一句话 | 选题门槛 G3 的判断依据 | AI / 科技 / 商业资讯整理，帮读者省时间 |
| 读者画像 | 相关度评分的锚点 | 普通科技关注者、AI 工具用户、创业者、内容从业者 |
| 选题偏好 / 雷区 | 偏好的主题、明确不碰的主题 | 偏好：AI 产品与工具、平台动态、政策监管；雷区：娱乐八卦、未证实爆料 |
| 栏目名 | 速览/头条/快讯等区块的自定义名称（账号资产） | 「本期速览」「今日重点」「值得一看」「一句话快讯」 |
| 语气参数 | 克制度、口语度（1 主 1 备风格的微调） | 克制为主，标准区可适度口语化 |
| 排版主色 | L2 排版稿的强调色 | #07C160（可改） |
| 结尾互动句式库 | 轮换使用，避免每期雷同 | 3 条克制版（见 11.4） |
| AI 声明文案 | 铁律三的文末声明 | 见 9.5 默认文案 |

## 七、工作流设计（6 步）

| Step | 做什么 | 输入 | 输出 | 判断标准 | 失败处理 |
|---|---|---|---|---|---|
| 1 接收解析 | 识别输入类型并读取 | 文本/链接/卡片 md | 原始素材集 | 是否有基本事件 | 信息太少→追问(≤3 问) |
| 2 选题评分+组稿 | 聚类→双门槛筛→三维评分→组稿方案 | 素材集+账号画像 | 候选清单（评分明细+组稿+落选去向） | 第八章选题模型 | **等用户勾选**（D4 不变） |
| 3 事实提取 | 联网读原文→关原文→填卡片 | 入选条目 | 事实卡片 | 能否答 5W | 标记缺口/待核实 |
| 4 风险判断 | 套版权+事实细则打分 | 事实卡片 | 风险标注 | 三条铁律 | 高风险→降级(D5) |
| 5 成稿+排版 | 按分层节奏模板写 L1，再转 L2 | 卡片+模板+画像 | 推文稿(md) + 排版稿(HTML) | 可读/合规/匹配定位 | 不达标→一句话事实 |
| 6 检查+变现 | 跑清单（含量化回查、AI 声明）+ 变现建议 | 推文稿 | 检查清单+变现建议 | 无 ❌ 项 | 有 ❌→不建议发，给改法 |

### 最终输出结构（默认 8 块）

```
【1. 选题清单与组稿方案】   ← 评分明细、封面条目、主线、落选理由与去向
【2. 标题备选(5/篇)】       ← 围绕封面条目；稳重/传播/简报/科技/克制
【3. 公众号正文(L1)】       ← 分层节奏 markdown
【4. 微信排版稿(L2)】       ← 内联样式 HTML，可直接粘贴；附粘贴后自查 3 点
【5. 来源与风险提示】       ← 哪些需以官方披露为准、哪些不可夸大
【6. 版权与合规检查】       ← 量化回查结果 + AI 声明确认 + 原创标确认
【7. 配图建议(文字版)】     ← 绝不用原文图
【8. 变现建议】             ← 低门槛、不污染可信度
```

## 八、选题评分与组稿设计（v2.1 新增，重点模块）

> 细则将落地到 `references/topic-selection.md`。口径：**双门槛一票否决 + 读者价值排序**（D8）。
> 上游卡片 ⭐️ 评分废弃不用（理由见 6.2）。

### 8.1 三道门槛（一票否决，不进入排序）

| 门槛 | 内容 | 不过槛的去向 |
|---|---|---|
| G1 版权门槛 | 来源分级（9.2）为 C 且无法回溯一手源；或必须大量依赖单篇表达才能成稿 | 降级简讯 或 弃用 |
| G2 事实门槛 | 核心事实无法用 ≥1 个 S/A 级来源确认 | 降级简讯（标"待核实"）或 弃用 |
| G3 定位门槛 | 与账号画像的定位/雷区冲突（如纯娱乐八卦） | 弃用 |

### 8.2 三维读者价值评分（过槛条目，每维 0–10）

**总分 = 相关度×0.4 + 话题度×0.3 + 信息增量×0.3，再乘时效系数。**

| 维度 | 问的问题 | 锚点 |
|---|---|---|
| 读者相关度（0.4） | 离读者的工作/生活/钱包/正在用的产品有多近？ | 9–10=几乎每个读者直接受影响（微信新功能）；5–6=行业相关者关注（公司融资）；1–2=远离日常（他国产业政策） |
| 话题度（0.3） | 自带讨论点/情绪点吗？读者会转发吗？ | 9–10=天然社交话题；5–6=圈内谈资；1–2=无人讨论 |
| 信息增量（0.3） | 读者从别处看饱了吗？我们能用名词卡/背景卡/对比给出增值吗？ | 9–10=别处没讲清+我们能讲清；5–6=有增值空间；1–2=全网刷屏且无差异化角度 |
| 时效系数 | 非政策类：>72 小时 ×0.8，>1 周 ×0.5；政策/法规类衰减放宽 | — |

**用 6/9 真实数据校准（锚点示例）**：

| 条目 | 上游⭐️ | 相关度 | 话题度 | 增量 | 总分 | 结论 |
|---|---|---|---|---|---|---|
| 微信小程序接入 AI | 7.0 | 9（人人用微信） | 8 | 7 | **8.2** | 封面条目候选 |
| 月之暗面融资破百亿美元 | 9.0 | 6 | 8 | 7 | **6.9** | 头条区候选 |
| 英国拟购本土 AI 芯片 | 6.0 | 3（离中文读者远） | 4 | 5 | **3.9** | 简讯或弃用 |

> 这正是"上游评分 ≠ 公众号价值"的实证：⭐️9.0 的融资新闻不如 ⭐️7.0 的微信功能贴读者。

### 8.3 组稿规则（一期合集不是 N 条并列）

1. **封面条目** = 总分最高者，决定主标题方向与封面配图建议方向。
2. **主线检测**：头部条目能否 2–3 条串成主题（按聚类标签判断，如"国产 AI 双响"）；有主线则导语点题。
3. **排列**：头条区（1–2 条）→ 标准区（3–4 条，同主题相邻）→ 简讯区（2–3 条，含降级条目）。
4. **条数**：默认一期 5–8 条（上限 10）。候选不足 5 条 → 建议改单条轻解读或并期发布。
5. **同质化检查**：与全网刷屏内容重合时，必须给出差异化角度（名词卡/背景卡/对比），给不出则降权。
6. **落选输出**：每条落选条目给理由 + 去向（简讯区 / 周报池留用 / 弃用）——不让用户猜。

### 8.4 与 D4 的关系（保持人工卡点）

评分明细 + 组稿方案 + 落选去向，整体作为候选清单交**用户勾选**；AI 不替用户拍板。
用户说"你来定"时，按本章模型自动定稿并注明。

## 九、版权风控设计（重点模块）

> 目标不是"看起来像原创"，而是**真的不侵权**。完整细则见 `references/copyright-guardrails.md`。

### 9.1 核心原理与纪律（v2.0 保留）

1. **事实 vs 表达**：事实（谁/何时/做了什么/金额/出处）不受版权保护；
   表达（遣词/句子/段落顺序/标题/配图）受保护。**从事实重写，绝不从表达重写。**
2. **反洗稿纪律**：开链接→抠事实记要点→关原文→凭笔记成稿→按 9.3 量化回查。
   "同义词替换式改写"仍是洗稿、仍侵权。
3. **七不照搬**：不搬标题/开头/段落结构/大段引用/同义词洗稿/原图/默认原创声明。
4. **引用边界**：短引用加引号+注明出处；客观事实陈述无需引号但要附来源；引用占比见 9.3。
5. **图片（v1）**：绝不用原文图，只给文字配图建议；数据图须用核实过的数据**重新制作**。

### 9.2 来源分级表（v2.1 新增，按级定权限）

| 级 | 典型来源 | 事实使用 | 表达使用 | 处理要求 |
|---|---|---|---|---|
| S 一手权威 | 政府/监管公告原文、法院公开文书、上市公司公告、公司官方 newsroom/博客、开源项目 release notes、官方技术文档 | 自由使用（注明出处） | 仅短引用加引号 | 优先选用；政策/司法类必须落到 S |
| A 多源权威 | ≥2 家权威媒体交叉一致的公共报道 | 可用 | 不照搬任何一家的表达 | 常规反洗稿纪律 |
| B 单源独家 | 单一权威媒体独家、专访、付费墙内容 | **谨慎**：只取核心事实，显性归因"据 XX 报道"；不展开独家细节链 | 禁止 | 见 9.4 独家规则 |
| C 二手转述 | 自媒体公众号（mp.weixin.qq.com）、Telegram 频道、聚合号、论坛、传闻 | 不当事实，仅当线索 | 禁止 | **必须回溯一手源**；回溯不到→降级"传闻"或弃用 |

微信特殊雷区随级别内化：C 级里的公众号来源，绝不改写那篇公众号本身（洗另一个公众号是平台
洗稿投诉打击最重的行为）；整篇转载需授权+白名单，本 skill 默认不做。

### 9.3 量化回查标准（v2.1 新增，"实质相似"的可执行版）

成稿后逐条执行，全部通过才算过版权关。**Phase 2 在应用层用代码实现**（字符串级比对），
把"人工盯"变成"代码盯"：

| # | 检查 | 阈值 / 规则 | 不过的处理 |
|---|---|---|---|
| Q1 | 连续重合 | 与任一原文**连续 ≥13 个汉字**相同（专有名词、加引号的直接引语除外） | 重写该句 |
| Q2 | 开头比对 | 成稿前 50 字与各原文开头无实质相似 | 重写开头 |
| Q3 | 结构比对 | 小标题/条目顺序不得与**单一来源**的段落结构一致（多源综合天然规避） | 重排结构 |
| Q4 | 引用占比 | 直接引语 ≤ 正文 10%，每处加引号+出处 | 删减引用 |

> 13 字阈值是**操作基线**（沿用中文查重惯例），不是法律标准；它的作用是把"自觉"变成"可检"。

### 9.4 独家内容处理规则（v2.1 新增）

"事实不受版权保护"只说对一半：**单篇独家报道的独家事实+细节链**（如独家专访拿到的数字），
大量取用即便不构成著作权侵权，也有**反不正当竞争**风险，且微信洗稿投诉照样成立。规则：

1. 单一独家大新闻：优先**等第二来源**（24–48 小时）再正常成稿。
2. 等不到且必须发：降级为一句话简讯，显性归因「据 XX 独家报道」，**只取最核心事实，不复述细节链**。
3. 付费墙内容：同上处理；绝不大段转述付费内容（既是版权问题也是对原媒体的实质性替代）。

### 9.5 AI 内容标识（v2.1 新增，铁律三细则）

依据《人工智能生成合成内容标识办法》（2025-09-01 施行）及微信公众平台配套规则：

1. **文末固定声明**（默认文案，可在账号画像里改）：
   > 本文由 AI 辅助整理自公开资讯，发布前已经人工核对；信息以各信源官方披露为准。
2. **平台勾选**：发布时使用公众平台的 AI 内容声明功能。
3. **不打"原创"标**：整理/编译类内容不属于法律意义的原创（与七不照搬第 7 条一致）。
4. 检查清单中设为**硬性 ❌ 项**：缺声明即不建议发布。

### 9.6 多源合并与风险评分（v2.0 保留）

- 同一事 ≥2 独立来源交叉，用自己的结构综合——既提可信度又远离单一原文表达。卡片"参考链接"区是现成多源入口。
- 版权风险评分：低=多源+注明+原创配图；中=单一来源/细节高度依赖某篇；高=单一自媒体无一手源/必须大量依赖原文表达。

## 十、事实与风险控制设计

> 完整细则见 `references/fact-risk-control.md`。

- **三分法**：事实（据公告/公开资料显示）｜推测（可能/或将/有望）｜观点（目前来看/一个值得关注的信号是）。
- **不确定信息**：据悉/传闻/知情人士 → 保留不确定性，建议补来源，或不收录。
- **高风险领域规则**：投融资（不预测收益/不荐买卖）、股价市值（标时间/不预测涨跌）、
  政策监管（引原文口径/不替代法律解释）、医疗（不做诊疗结论）、法律（用"涉嫌/争议/有待确认"）、
  金融（禁"稳赚/必涨/抄底"）。
- **联网核对触发**：关键金额/轮次/投资方、政策条款、判决罪名刑期、最新股价、所有"据悉/传闻"。

## 十一、内容与输出形态设计（v2.1 重写）

> 模板与示例落地到 `references/digest-templates.md`；排版规则落地到 `references/wechat-formatting.md`。

### 11.1 三种形态（D3 不变）

- **模板 A · 资讯速递合集（默认）**：v2.1 升级为分层节奏版（11.2）。
- **模板 B · 单条轻解读**：800–1500 字，四段式展开 + 名词卡/背景卡。
- **模板 C · 工具/公司动态简报**：同类资讯聚成专题。

### 11.2 模板 A v2 · 分层节奏（D10）

替代"每条统一四段式"。结构与字数配比：

```
【本期速览】     3–5 行，每行一句话一条（目录式，方便读者决定读不读）
【头条区】1–2 条  标题 + 发生了什么 + 关键信息 + 名词卡/背景卡 + 为什么值得关注
                 +（仍需注意，条件字段）+ 来源；每条 300–500 字
【标准区】3–4 条  四段式：发生了什么 / 关键信息 / 为什么值得关注 /（仍需注意，条件字段）
                 + 来源；每条 100–200 字
【简讯区】2–3 条  一句话 + 来源或"传闻，待核实"标注（D5 降级条目落在这里）
【结尾区】       克制收束一句 + 互动引导（从画像句式库轮换）+ AI 声明（9.5）+ 参考来源汇总
```

关键变化：

- **"仍需注意"条件化**：只在存在待核实数据/传闻/高风险领域时出现。每条都挂"以官方为准"
  是免责声明滥用，反而显得不专业。
- **区块名用账号画像里的栏目名**（如「今日重点」「一句话快讯」），让结构成为账号资产。
- 各区字数配比即"节奏"：深一条、平几条、快几条，读者不腻。

### 11.3 轻度解读的三个安全角度（"为什么值得关注"的写法升级）

按优先级使用，全部不依赖行业判断力、全部可核实：

1. **影响角度**：对"正在用 X 的人"意味着什么（基于已证实的功能/价格/范围）。
2. **连线角度**：与此前已证实事件连成时间线（"这是该公司今年第二次…"→ 背景卡）。
3. **对比角度**：与同类可比对象的公开参数/价格对比（→ 名词卡/对比）。

写不出以上三种，就只留事实，不硬凑观点。禁用判断类句式（"必将颠覆""格局已变"）不变。

### 11.4 结尾互动句式库（默认 3 条，画像可改）

- 「这几条里你最关心哪条？评论区聊聊。」
- 「觉得有用可以转给同样关注 AI 的朋友。」
- 「我们每天帮你筛一遍资讯，省下刷信息流的时间——点个关注不迷路。」

### 11.5 微信排版稿 L2（D9 新增交付物）

**生成方式**：模型只产 L1 markdown；L2 由确定性转换产生——
Phase 1 调用现成 md→微信 HTML 转换工具链（环境内已有同类 skill 可用），Phase 2 用应用层转换器。

**微信编辑器约束（必须遵守，否则粘贴即坏）**：

1. 全部样式**内联 style**（编辑器会剥离外部 CSS / class / JS / iframe / position 定位）。
2. **正文外链不可点**：非微信生态链接在正文中显示为纯文本「媒体名+日期」；
   完整链接集中放结尾区「参考来源」列表；最重要的一条可建议设为"阅读原文"。
3. 图片只放占位说明（配图仍走 D6 文字建议）。

**默认样式 token（画像可改主色）**：

| 元素 | 样式 |
|---|---|
| 正文 | 15px / 行高 1.75 / 段后距 16px |
| 头条区标题 | 17px 加粗，主色左边框或下划线 |
| 区块标题 | 16px 加粗 + 主色（默认 #07C160） |
| 简讯区 | 14px 灰（#888） |
| 来源/声明行 | 13px 浅灰（#999） |

**粘贴后自查 3 点**（随 L2 一起输出）：样式是否保留、有无乱码、来源是否按"正文纯文本+文末列表"呈现。

### 11.6 标题（v2.0 保留 + 封面条目导向）

每篇 5 个备选（稳重/传播/简报/科技/克制），不照搬原标题、不标题党；
**主标题围绕封面条目（8.3）展开**，速览式标题可带"｜本期 N 条"后缀。

## 十三、变现设计

> 细则见 `references/monetization.md`。底线：**变现不得损害可信度**。

- 方式：流量主、文中广告、互选广告、AI 工具分销、资料包引流、社群引流、课程分销、暂不适合。
- 速递合集现实：最稳的是**流量主 + 结尾自然引导关注/收藏/转发**；严肃政策/法律/医疗稿不插硬广。
- 必给：推荐方式、是否插广告、是否引导关注/领资料、**是否影响可信度**（必须明说）。

## 十四、部署方案（两阶段，详细）

### 14.1 Phase 1 · Claude / Codex 原生技能（先做）

- **形态**：`wechat-news-digest/`（SKILL.md + references），可用 skill-creator 打包成 `.skill` 安装。
- **调用**：对话中触发；用户给资讯/链接/卡片 md → skill 按 6 步走 → 关键卡点（选题勾选、发布前检查）人工确认。
- **L2 排版稿**：成稿后调用现成 md→微信 HTML 转换工具链生成；无工具时按 `wechat-formatting.md` 的
  token 表手动转换（仍是确定性映射，不让模型自由发挥 HTML）。
- **优势**：最快可用、最易迭代、联网读原文可用。

### 14.2 Phase 2 · 自建网页 + 第三方大模型 API

网页端没有技能加载机制，需要把 skill"编译"成可注入的提示 + 由应用编排。两种实现：

- **方案 A · 编排式（推荐，生产）**：网页按 6 步发多次 API 调用，每步只注入相关 reference 拼成的系统提示。
  - Step 2 选题、Step 6 检查作为**网页 UI 卡点**（人工勾选/确认），天然实现 D4/D5；
    选题评分明细（第八章）直接渲染成 UI 表格供勾选。
  - 每次调用聚焦、更便宜、规则遵循更稳。建议用 **prompt caching** 缓存较大的系统提示段。
- **方案 B · 单大提示（MVP）**：把 SKILL.md + 全部 references 拼成一个大系统提示，一次调用产出全部。
  易搭但更贵、人工卡点不好插、规则遵循易漏。适合先跑通 demo。
- **两个应用层确定性组件（v2.1 新增，不耗 tokens、不依赖模型自觉）**：
  1. **版权量化检查器**：实现 9.3 的 Q1–Q4（字符串/最长公共子串级比对）——API 端没人盯着，
     这是把版权风控从"模型自觉"变成"代码强制"的关键。
  2. **md→微信 HTML 转换器**：实现 11.5 的固定映射与样式 token。
- **离线降级（重要）**：API 端通常不能联网 → 把 D2「联网读原文」降级为「只吃事实卡片 + 凡未证实一律标待核实」。
  这要求**上游抓取流程**（产出 6/9 那种 md）作为依赖先就位。
- **配置注入**：账号画像（6.4）作为请求级配置 JSON 注入，支持未来多账号复用同一套编排。
- **交付物**：「系统提示包」（由 skill 文件编译而来）+ 上述两个组件的实现说明。
- **注意**：模型选择（建议用能力强的模型保事实与合规）、上下文窗口、单篇成本、并发与重试。
